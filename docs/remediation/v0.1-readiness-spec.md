# Spec: fechamento de prontidao v0.1

## Tipo do artefato

human documentation / execution spec

## Finalidade

Definir a execucao necessaria para transformar o estado `CONDITIONAL` em `PASS` para v0.1.

Este spec aplica as correcoes pendentes da auditoria sem desviar a filosofia do repositorio:

- baixa carga cognitiva
- fonte primaria unica
- `find -> select -> inject`
- guardrails robustos sem travar tarefas seguras
- documentacao humana explicativa, fora da composicao padrao

---

## Estado atual

O repositorio esta bloqueado para v0.1 por:

- `ORQ-06`: duplicidade e excesso de exemplos no cluster de naming.
- `ORQ-07`: prompts e skills ainda carregam exemplos longos que deveriam estar em docs ou evals.
- `evals/`: suite criada, mas ainda nao executada e registrada.

---

## Objetivo

Concluir a v0.1 com:

- cluster de naming mais compacto
- prompts e skills menores
- exemplos longos movidos para documentacao ou evals
- suite minima de evals executada
- registro final de release readiness

---

## Nao objetivos

Nao fazer nesta execucao:

- criar ferramenta automatizada de eval
- criar `tasks/`, `workflows/` ou `policies/`
- criar template por categoria
- reescrever toda a taxonomia
- introduzir dependencia de fornecedor/modelo
- mover `docs/` ou `evals/` para contexto padrao

## Regras de refatoracao cognitiva

### Prompts

Prompts devem conter:

- objetivo
- entradas obrigatorias
- composicao recomendada
- instrucao operacional
- saida esperada
- checkpoints

Prompts nao devem conter:

- exemplos longos
- tabelas extensas
- regra primaria
- procedimento tecnico detalhado

### Skills

Skills devem conter:

- procedimento reutilizavel
- pre-condicoes
- limites
- referencias para rules

Skills nao devem conter:

- matriz normativa completa
- severidade primaria
- exemplos extensos
- politica de governanca

### Rules

Rules devem conter:

- obrigacao
- proibicao
- criterio de conformidade
- severidade, quando aplicavel

Rules nao devem conter:

- tutorial operacional longo
- fluxo completo de prompt
- exemplos redundantes em todos os arquivos

### Docs e evals

Docs podem conter exemplos explicativos.

Evals podem conter exemplos de entrada, saida esperada e criterio de aceite.

Nenhum dos dois entra na composicao padrao.

---

## Criterios de aceite v0.1

v0.1 so pode ser liberada quando:

- `ORQ-06` estiver `PASS`.
- `ORQ-07` estiver `PASS`.
- Os 10 evals tiverem resultado registrado.
- Nenhum eval bloqueante estiver `FAIL`.
- Prompts de naming nao contiverem exemplo completo longo.
- Skills de naming nao duplicarem matriz normativa primaria.
- `rules/naming/reference-severity.md` continuar sendo unica fonte primaria de severidade.
- `rules/naming/reference-units.md` continuar sendo unica fonte primaria de unidades.
- `git diff --check` passar.
- Nenhum diretorio versionado ficar sem `README.md`.
- O registro final declarar `Status geral: PASS`.

## Riscos

| Risco | Impacto | Mitigacao |
|---|---|---|
| Compactar demais e perder instrucao operacional | Agente improvisa | Manter contrato minimo em prompts e skills |
| Mover regra para docs por engano | Docs vira fonte normativa | Toda regra deve permanecer em `rules/` |
| Evals virarem nova governanca | Taxonomia paralela | Evals validam comportamento, nao definem norma |
| Refatorar naming inteiro | Churn alto | Atuar somente em duplicidade e exemplos longos |
| Baixar guardrail para reduzir atrito | Risco operacional | Manter `operational-safety-guardrails.md` intacto |

---

## Veredito esperado

